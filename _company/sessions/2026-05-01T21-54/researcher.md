## 📘 Part 1: 표준 지식 그래프 (KG) 매핑 구조 제안
**(정보 자산 및 기술적 의사결정 과정 통합용)**

모든 에이전트가 생성하는 산출물을 단순 데이터가 아닌, **상호 연결된 '지능형 프로세스 시스템'의 지식 파편**으로 취급하기 위해 다음 3가지 컴포넌트를 기본 골격으로 합니다. 이 구조는 EPHA를 단순 앱이 아닌, 문제 해결 방법론을 자산화하는 플랫폼으로 포지셔닝합니다.

### 1. 노드 (Node) 정의: 정보의 최소 단위
노드는 그래프 내에서 독립적인 지식 객체(Entity)를 나타냅니다. 모든 노드는 반드시 고유 식별자(ID)와 타입 메타데이터를 가져야 합니다.

| 필수 필드 | 데이터 타입 | 설명 및 용도 | 예시 (실제 값) |
| :---: | :---: | :--- | :--- |
| **`Node_ID`** | String (UUID v4) | 노드의 고유 식별자. 중복 절대 불가. | `KG-20260501-A7B3D` |
| **`Node_Type`** | Enum/String | 지식의 근본 유형 (분류). 시스템이 검색할 때 가장 먼저 필터링하는 기준. | `[Concept], [Technology], [Process], [Case Study]` |
| **`Title`** | String | 노드의 핵심 주제 또는 이름. 간결하고 직관적이어야 함. | 'Adapter Layer', 'KPI 정의 방법론' |
| **`Content_Summary`** | Text | 2~3줄로 요약된 핵심 지식 파편. (불필요한 배경 설명 제거) | "A와 B 사이의 비동기 통신을 가능하게 하는 추상화 계층." |
| **`Source_Link`** | URL/Path | 해당 정보를 얻은 원본 자료의 링크 또는 내부 폴더 경로. **(사실 검증 필수)** | `sessions/2026-05-01T14:36/researcher.md` |

### 2. 엣지 (Edge) 정의: 지식 간의 관계성
엣지는 노드와 노드를 연결하는 관계성(Relationship)을 나타냅니다. 단순한 '관련'이 아닌, **구체적인 상호작용**을 명시해야 합니다.

| 필수 필드 | 데이터 타입 | 설명 및 용도 | 예시 (관계 정의) |
| :---: | :---: | :--- | :--- |
| **`Edge_ID`** | String (UUID v4) | 엣지의 고유 식별자. | `REL-A7B3D-TO-E1F2G` |
| **`Source_Node_ID`** | String | 시작 노드의 ID. | `KG-20260501-A7B3D` (Adapter Layer) |
| **`Target_Node_ID`** | String | 끝 노드의 ID. | `KG-20260501-E9F0G` (Isolation 원칙) |
| **`Relationship_Type`** | Enum/String | 관계의 종류를 정의. 이 부분이 '지능형' 핵심입니다. | `[INFLUENCES], [IS_A_SOLUTION_FOR], [REQUIRES], [DEVELOPS]` |
| **`Context_Explanation`** | Text | 왜 이 두 노드가 연결되었는지 설명하는 논리적 근거. (가장 중요) | "Adapter Layer는 Isolation 원칙을 유지하기 위해 필수적으로 A에 영향을 준다." |

### 3. 매핑 예시 (지식의 흐름):
*   **[Process]** `MVP 개발` $\xrightarrow{\text{REQUIRES}}$ **[Concept]** `KPI 확정` $\rightarrow$ **[Technology]** `Connection Storytelling`
