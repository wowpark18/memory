## 📚 Connection Storytelling 알고리즘 테스트용 모의 데이터 세트 (Mock Data Set)

**[데이터 구조 원칙]**
*   **Node ID:** 모든 정보 조각에 고유 식별자 부여 (필수).
*   **Type Metadata:** 해당 정보가 어떤 종류의 지식인지 정의 (Concept, Tool, Problem, Principle 등).
*   **Connection Flow:** 데이터들이 어떻게 연결되어야 하는지를 `[Source ID] -> [Target ID]` 형태로 명시하여 알고리즘의 테스트 케이스를 제공합니다.

| :--- | :--- | :--- | :--- | :--- |
| **문제 정의** | `P_001` | Problem | 외부 API를 내부 시스템에 통합할 때 발생하는 데이터 구조 불일치 문제. | 핵심 Pain Point 설정. |
| **원리 지식** | `C_A01` | Principle | Adapter Layer: 서로 다른 인터페이스를 가진 컴포넌트 간의 통신을 중재하는 계층적 패턴. | 해결책의 근본 원칙 제시. |
| **실패 사례** | `S_L01` | Failure Log | 이전 버전 코드가 강한 결합도(Tight Coupling)로 인해 업데이트된 API 요청 형식에 대응하지 못함. | 실패 과정을 구체화. |

| **Connection Flow** | `P_001` $\to$ `S_L01` $\to$ `C_A01` (문제 인식) $\to$ `D_A02` (정보 탐색) $\to$ `K_C03` (해결 적용) |

### 💡 Mock Data Set 2: MVP 범위 확장으로 인한 기술 부채 누적
| 항목 | Node ID | Type Metadata | 내용 / 값 | 역할/설명 |
| :--- | :--- | :--- | :--- | :--- |

| **원리 지식** | `C_P01` | Principle | MVP(Minimum Viable Product): 최소 기능으로 핵심 가치를 검증하는 데 초점을 맞춘다. | 전략적 해결책 원칙 제시. |
| **관련 논의** | `D_V01` | Document | '정보 단절 해소' 프로세스의 정의: 단순 검색이 아닌, 지식 연결성 확보가 목표여야 함. | 핵심 가치 상기 자료. |
| **위험 요소** | `R_T01` | Risk Assessment | 복잡한 기능 추가는 테스트 가능성을 낮추고 개발 기간을 무한히 늘릴 수 있음. | 문제의 심각성을 경고하는 정보. |
| **최적화 방안** | `K_A03` | Action Plan | '핵심 지능 로직'에만 집중하고, 나머지 기능은 V2로 분리하여 추후 개발한다. | 구체적인 실행 계획 제시. |
| **Connection Flow** | `P_002` $\to$ `C_P01` (원칙 확인) $\to$ `D_V01` (가치 재정립) $\to$ `R_T01` (위험 분석) $\to$ `K_A03` (방향성 확정) |
