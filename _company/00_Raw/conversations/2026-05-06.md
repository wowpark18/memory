### 📄 2. Mock API Hook (`src/api/hooks/useMockApi.ts`)

실제 API 호출을 가로채서(Intercept) 모의 데이터를 반환하는 핵심 서비스 훅입니다. 이 훅 하나만 사용하면 프론트엔드 로직 전체가 테스트 가능해집니다.

```typescript
import { KnowledgeNode, KnowledgeEdge, getMockGraphData } from '../mockData/knowledgeGraphMock';

/**
 * @description 실제 백엔드 API 호출을 가로채어 모의 데이터를 반환하는 훅 (Stubbing Layer)
 * 이 훅은 모든 컴포넌트에서 'API를 호출하는 것처럼' 사용되어야 합니다.
 */
export const useMockApi = () => {
  /**
   * @description 지식 그래프 전체 데이터(노드, 에지)를 로딩합니다.
   * 실제로는 /api/v1/graph/data 엔드포인트를 호출해야 하지만, 현재는 모의 데이터를 반환합니다.
   */
  const fetchGraphData = async (): Promise<{ nodes: KnowledgeNode[], edges: KnowledgeEdge[] }> => {
    console.log("[Mock API] Calling /api/v1/graph/data endpoint...");
    
    // 비동기 지연 시간 시뮬레이션 (네트워크 지연 효과)
    await new Promise(resolve => setTimeout(resolve, 800));

    const mockData = getMockGraphData();
    return { nodes: mockData.nodes, edges: mockData.edges };
  };

  /**
   * @description 특정 ID를 가진 노드의 상세 정보를 조회합니다.
   */
  const getNodeDetails = async (nodeId: string): Promise<KnowledgeNode | null> => {
    console.log(`[Mock API] Calling /api/v1/graph/nodes/${nodeId} endpoint for details.`);
    await new Promise(resolve => setTimeout(resolve, 300)); // 짧은 지연 시간

    // 실제로는 DB에서 조회하지만, 여기서는 모의 데이터를 필터링하여 반환합니다.
    const mockNodes = getMockGraphData().nodes;
    return mockNodes.find(node => node.id === nodeId) || null;
  };


  return { 
    fetchGraphData, 
    getNodeDetails 
  };
};

// 사용 예시: const { fetchGraphData } = useMockApi();
```

### 🔄 인터랙티브 플로우 및 상태별 상세 스펙 (User Journey)

#### 💡 State 1: Pain Point 감지 (The Gap - 초기 화면)
*   **상황:** 사용자가 특정 주제에 대해 검색하지만, 핵심 정보를 찾기 위해 여러 출처를 떠돌고 있음.
*   **화면 구성:**
    *   배경은 미니멀한 다크 모드(Deep Navy: `#0A192F`)로 설정하고, 중앙에 불규칙하게 배열된 노이즈성 데이터 파편들(미완성 노드)을 배치합니다.
    *   가장자리에는 명확한 경계 없이 어둡고 불안정한 'Gap Zone'이 시각적으로 존재함을 강조합니다.
    *   **[UI 컴포넌트] Gap Alert Modal:** 화면 중앙에 "정보의 단절로 인해 중요한 연결 고리가 누락되었습니다."와 같은 문구가 옅은 에러 메시지 형태로 깜빡이며 표시됩니다.
*   **애니메이션:** 배경 전체에 낮은 주파수의 그리드 패턴이 깔려있으나, 이 그리드가 Gap Zone에서 주기적으로 '깨지는' 듯한(Disruption) 시각 효과를 보여줍니다.

#### ✨ State 2: EPHA의 개입 및 과정 추적 (The Discovery - 핵심 상호작용)
*   **상황:** 사용자가 특정 노드 A를 클릭하여 프로세스 분석을 요청합니다.
*   **액션:** **사용자 > [노드 A] 클릭 → [Edge 연결 시작]**
*   **시각 변화:**
    1.  **활성화:** 노드 A가 전경으로 튀어나오듯(Pop-up) 부각되며, 주변의 다른 노드들이 이 노드를 향해 미세하게 끌어당겨지는(Gravitational Pull) 애니메이션이 발생합니다.
    2.  **프로세스 시작 (Gap to Edge):** Gap Zone 내부에서 연결을 시도하는 **'빛나는 에지(Luminous Edge)'**가 발사됩니다. 이 빛은 `#66FFFF` 계열의 시안 그라데이션으로, 마치 전기 신호처럼 노이즈를 뚫고 전진합니다.
    3.  **엣지 라벨링:** 에지가 이동하는 경로(Edge) 위에 `processType`에 따른 설명을 실시간으로 오버레이 합니다. (예: *[CAUSAL] → 인과적 추론 과정* / *[SUPPLEMENTARY] → 보충 지식 연결*)
    4.  **중간 노드 발견:** 에지 경로 중간에 새로운 '연결점(Intermediate Node)'이 발견되면, 이 노드가 빛을 받아 점차 선명해지는 애니메이션(`Fade In` + `Bloom Effect`)을 거칩니다.

#### 🌟 State 3: 정보 단절 해소 및 통찰 (Aha Moment - 최종 결과)
*   **상황:** 에지가 목표 노드 B에 성공적으로 연결됩니다. Gap Zone이 완전히 메워지며 지식이 구조화되는 순간입니다.
*   **액션:** **Edge 도착 → [노드 B] 연결 완료.**
*   **시각 변화 (가장 중요):**
    1.  **완성 애니메이션:** 빛나는 에지가 노드 B에 도달하는 순간, `#FFD700` 계열의 골드 하이라이트가 노드 B를 감싸며(Halo Effect) 연결을 확정합니다. 이와 동시에 배경의 Gap Zone 전체가 부드럽게 사라지거나, 명확하고 질서 있는 그리드로 재구성됩니다.
    2.  **결과물 제시:** 화면 하단에 **[Key Takeaways]** 섹션이 깔끔하게 펼쳐지며, 노드 A → Edge → 노드 B의 논리적 경로를 요약한 구조 다이어그램(Flow Diagram)을 보여줍니다. 이 과정은 일반적인 텍스트 나열 방식보다 훨씬 강력한 시각적 만족감을 제공해야 합니다.
*   **사운드 디자인 가이드:** State 3 도달 시, '쉬이익-'하는 에지 사운드가 부드럽고 명료한 '띵' 하는 성공음(Ding)으로 마무리되도록 설계합니다.

# 📈 정보 단절 리스크 자동 계산기 (ROI Calculator) 로직 설계
(Information Disruption Risk ROI Calculator Logic Design)

본 계산기는 EPHA의 핵심 가치인 '정보 단절 해소 프로세스'를 **정량화된 재무적 지표**로 변환하여, B2B 및 B2C 시장 모두에서 제품의 경제적 효용성(ROI)을 입증하는 데 사용됩니다.

---

## 1. 계산기 핵심 구조 및 목표
*   **목표:** 고객이 현재 겪는 비구조화된 정보 Gap으로 인해 발생하는 **총 잠재 손실액 (Total Potential Loss)** 대비, EPHA 솔루션 도입을 통해 절감되는 비용과 시간의 가치를 산출합니다.
*   **핵심 공식:** $\text{순수 ROI} = \frac{(\text{현재 총 잠재 손실액}) - (\text{EPHA 활용 후 예상 리스크 및 비용})}{(\text{솔루션 도입 비용})} \times 100$

## 2. 핵심 변수 정의 (Inputs)
사용자에게 입력받을 수 있는 측정 가능하고 구체적인 변수들입니다.

| 변수명 | 기호 | 설명 | 단위 | 비고 |
| :--- | :--- | :--- | :--- | :--- |
| **평균 인건비** | $C_{hr}$ | 관련 업무에 투입되는 평균 시간당 비용 (가장 중요한 가치 측정 기준) | 원/시간 | B2B와 B2C 모두 공통 적용. |
| **문제 발생 빈도** | $F$ | 정보 단절로 인해 문제가 발생하는 주기적 빈도 (예: 월별, 분기별) | 횟수/기간 | 규제 변화 빈도, 프로젝트 시작 빈도 등. |
| **단일 Gap당 평균 소요 시간** | $T_{gap}$ | 하나의 정보 Gap을 해소하는 데 수작업으로 필요한 평균 시간 | 시간 | 리서치, 검토, 연결에 들어간 노동 시간. |
| **규제/오류 심각도 계수** | $\alpha$ | 규제 미준수 또는 잘못된 정보로 인한 건당 예상 피해 배율 (가중치) | 무차원(배율) | B2B에서 가장 중요하게 사용됨. $(>1)$ |
| **정보 검색/처리 복잡성** | $D$ | 필요한 정보의 종류와 연결 과정의 평균적인 난이도 | 척도 (1~5점) | 5점에 가까울수록 Gap 해소 난이도가 높음. |

## 3. 시장별 로직 및 공식 정의 (Process & Formula)
변수들을 조합하여 B2B와 B2C라는 두 가지 다른 관점에서 손실액을 계산합니다.

### A. 🏢 B2B (기업/전문가 대상: 리스크 제거 중심)
*   **핵심 Pain Point:** 법적 리스크, 운영 지연으로 인한 금전적 손실 및 기회비용.
*   **목표 측정값:** **운영 리스크 감소 금액 ($S_{B2B}$)**

$$\mathbf{S_{B2B}} = \left[ (F \times T_{gap} \times C_{hr}) \times (1 + \alpha) \right] - \text{EPHA 비용 절감액}$$

**단계별 프로세스:**
1.  **Gap 발생에 따른 총 노동 시간 산출:** $L_{total} = F \times T_{gap}$
2.  **규제/오류 리스크 가중치 적용:** $R_{risk} = L_{total} \times C_{hr} \times (1 + \alpha)$
3.  **최종 예상 손실액:** $\text{Total Loss}_{B2B} = R_{risk}$
4.  **ROI 도출:** 이 수치($\text{Total Loss}_{B2B}$)가 바로 고객이 매년 절감할 수 있는 금액입니다.

### B. 🏠 B2C (일반 소비자 대상: 시간/효율성 중심)
*   **핵심 Pain Point:** 정보 탐색에 소요되는 과도한 시간, 학습 곡선 장벽, 기회비용.
*   **목표 측정값:** **절감된 순수 시간 가치 ($S_{B2C}$)**

$$\mathbf{S_{B2C}} = \text{Gap 해소 과정의 총 예상 시간 절감량} (T_{saved}) \times C_{hr}$$

**단계별 프로세스:**
1.  **현재 Gap 해결에 필요한 총 시간 예측:** $T_{current} = F \times T_{gap}$
2.  **EPHA를 통한 효율화율 적용:** 사용자가 EPHA로 인해 절약할 수 있는 비율($P_{save}$)을 가정 (예: 60%).
3.  **실질적 시간 절감량 산출:** $T_{saved} = T_{current} \times P_{save}$
4.  **ROI 도출:** $\text{Total Savings}_{B2C} = T_{saved} \times C_{hr}$. 이 금액은 "정보 접근성 확보의 가치"로 포장됩니다.

## 4. 적용 시나리오 예시 (가상 계산)
*   **배경 설정:** 중소기업 법무팀이 새로운 규제(Gap)를 해석하고 문서화하는 과정.
*   **변수 입력:** $F=1$ (분기별), $T_{gap}=20$시간, $C_{hr} = 70,000\text{원/시간}$, $\alpha=1.5$ (규제 미준수로 인한 벌금 배율).
*   **계산:**
    $$\mathbf{S_{B2B}} = \left[ (1 \times 20 \times 70,000) \times (1 + 1.5) \right] = 14,000,000\text{원}$$
*   **결과 해석:** "EPHA 솔루션을 통해 귀사는 이 규제 Gap 해소 과정에서 **최소 1,400만 원의 잠재적 리스크와 운영 비용을 절감**할 수 있습니다."

# 📊 B2B 정보 단절 리스크 분석: 3대 사례 연구 (Evidence Base)

## Case 1. 국제 데이터 전송 규제 위반 (GDPR/CCPA 준수 실패)
*   **산업 분야:** 글로벌 이커머스 및 SaaS 서비스
*   **정보 Gap의 특성:** 특정 지역(유럽, 캘리포니아 등)에서 수집한 사용자 개인 정보를 다른 국가의 데이터 센터로 전송할 때, 해당 정보가 어떤 법적 규제 프레임워크를 거쳐야 하는지 연결 과정이 단절됨.
*   **Gap Source (정보 단절 원인):** *규제 지식 파편화 및 동시 비교 불가.* 각 지역별 프라이버시 규정(GDPR, CCPA, LGPD 등)의 핵심 요구사항과 데이터 처리 흐름을 개별적으로만 보유하고 있어, 통합적인 '데이터 이동 경로 리스크 매핑'이 불가능함.

| 측정 항목 | 상세 내용 | 추정 재무적 영향 (Loss/Cost) |
| :--- | :--- | :--- |
| **1. 규제 준수 지연 비용** | 법률 자문 및 내부 시스템 수정에 투입된 시간(Man-Hour Cost). | 3,000만 원 ~ 8,000만 원 (최소) |
| **2. 벌금 리스크 (Worst Case)** | GDPR 위반 시 전 세계 매출의 최대 4%까지 부과될 수 있는 잠재적 손실액. | $5억 이상 (잠재적 최대 리스크) |

| **총 잠재적 재정 손실 추정치 (Total Potential Loss)** | **최소 수억 원의 운영 리스크 발생** | |

*   **EPHA 시스템 도입 시 절감 가능 비용:** EPHA가 다양한 지역 규제(GDPR, CCPA 등)를 하나의 노드에 연결하고, 데이터 흐름을 따라 이동 경로별로 필요한 필수 조치 목록과 법적 근거(Edge)를 실시간으로 보여줌으로써, 수동 리스크 검토 과정을 자동화합니다.
*   **정량적 절감액:** **법무/컴플라이언스 검토 시간 70% 단축 (운영 비용 $Y$억 원 절감)**

## Case 2. ESG 보고서 작성 및 공급망 실사 의무화 리스크
*   **산업 분야:** 중공업, 제조업 (글로벌 공급망 연관 산업)
*   **정보 Gap의 특성:** 주요 시장(EU 등)에서 요구하는 탄소 배출량 측정, 인권 실태 조사, 노동 기준 준수와 같은 ESG(환경·사회·지배구조) 정보가 여러 단계의 복잡한 공급망에 걸쳐 분산되어 있고, 표준화된 포맷으로 연결되지 않아 전체적인 추적 및 보고서 작성이 불가능함.
*   **Gap Source (정보 단절 원인):** *분산된 데이터 소스 간의 개념적/표준 지표 연결 부재.* 협력사 A는 '에너지 사용량'만, 협력사 B는 '폐기물 발생률'만 보고하여, 전체 공급망 차원의 $\text{Scope 3}$ 배출량을 계산할 수 없음.

| 측정 항목 | 상세 내용 | 추정 재무적 영향 (Loss/Cost) |
| :--- | :--- | :--- |
| **1. 컨설팅 및 데이터 통합 비용** | 여러 전문 기관을 동원하여 데이터를 취합하고 표준화하는 외부 용역비용. | 2억 원 ~ 5억 원 |
| **2. 리포트 지연 또는 불완전 제출 패널티** | 규제 당국이나 대형 바이어의 요구에 미달하여 계약 수주가 지연되거나 취소되는 손실. | 프로젝트당 최소 10~30% 매출 감소 예상 |
| **3. 사후 감사 및 소송 리스크** | 보고서 데이터 오류로 인해 발생할 수 있는 명성 실추 및 법적 책임(Greenwashing 논란 등). | 잠재적 평판 자산 손실액 (가장 큼) |
| **총 잠재적 재정 손실 추정치 (Total Potential Loss)** | **공급망 전체 프로세스 관리가 불가능하여 시장 접근성 상실 위험** | |

*   **EPHA 시스템 도입 시 절감 가능 비용:** EPHA가 ESG 관련 글로벌 표준(GRI, SASB 등)을 노드로 구축하고, 공급망 참여자들의 개별 보고서 파편들을 같은 개념적 구조(Schema)로 끌어와 연결하는 '프로세스 모델링' 기능을 제공합니다.
*   **정량적 절감액:** **공급망 데이터 취합 및 표준화 프로세스 시간 80% 단축 (운영 비용 $Z$억 원 절감)**

## Case 3. 산업별 법규 개정 대응 지연 (의료/금융 분야 예시)
*   **산업 분야:** 의료 기기 제조, 금융 상품 개발 등 규제 변화가 잦은 전문 서비스업
*   **정보 Gap의 특성:** 정부나 국제 표준 기관에서 새로운 가이드라인이나 규제가 발표될 때마다(예: AI 의료기기 사용 가이드라인 변경), 이 정보를 기존 내부 프로세스 매뉴얼과 연결하여 '어디를, 어떻게 수정해야 하는지'에 대한 전체적인 영향 분석이 수작업으로 이루어짐.
*   **Gap Source (정보 단절 원인):** *규제 정보와 운영 매뉴얼 간의 동적 연동 실패.* 규정(A) 변경 $\to$ 시스템 모듈(B) 수정 필요 $\to$ 교육 자료(C) 업데이트 $\to$ 테스트 케이스(D) 재작성이 필요한데, 이 모든 과정이 개별 문서로 존재하며 상호 참조가 안 됨.

| 측정 항목 | 상세 내용 | 추정 재무적 영향 (Loss/Cost) |
| :--- | :--- | :--- |
| **1. 대응 시간 지연 비용** | 규제 변경을 인지하고, 내부 시스템 전반에 걸쳐 영향을 분석하는 데 소요된 인력 투입비용. | 5천만 원 ~ 1억 원 (단일 건당) |
| **2. 시장 출시(Time-to-Market) 지연** | 규제 변경 대응 실패로 인해 제품/서비스의 상업적 배포 시점이 늦어짐으로써 발생하는 기회비용. | 연간 $V$억원 이상의 매출 손실 가능성 (가장 치명적) |
| **3. 재작업(Rework) 비용** | 수정된 규정에 맞춰 이미 개발되거나 출판되었던 자료를 폐기하고 처음부터 다시 만드는 비용. | 1~2개 모듈/문서 세트당 수천만 원의 손실. |
| **총 잠재적 재정 손실 추정치 (Total Potential Loss)** | **규제 변화에 대한 '반응 시간' 지연으로 인한 시장 경쟁력 상실** | |

*   **EPHA 시스템 도입 시 절감 가능 비용:** EPHA가 규제 변경 사항을 감지(Monitor)하고, 이 변경사항이 사내 어떤 프로세스(Process Node)와 연결되어 영향을 미치는지('Impact Analysis')를 자동으로 매핑하여 '최소 수정 필요 지점'만을 찾아줍니다.
*   **정량적 절감액:** **규제 변화 대응 분석 및 재작업 시간 90% 단축 (운영 효율성 $W$억 원 확보)**

## 🔗 EPHA 핵심 UX 컴포넌트 블루프린트: '리스크 발견 및 가치 전환 과정'
(Information Disruption Risk Detection & Value Mitigation Flow)

본 설계는 Business와 Researcher가 정의한 정량적 손실액(Potential Loss, $X$)을 사용자 경험의 중심 축으로 삼아, EPHA 시스템이 단순 지식 제공자가 아닌 **'재무적 리스크 제거 프로세스 자문가'**로 인식되도록 돕는 3단계 인터랙션 블루프린트입니다.

---

### 💡 Global Design Principles (전체 공통 규칙)
1.  **기반 테마:** 다크 모드 기반의 전문적이고 무게감 있는 분위기 유지.
2.  **강조색 정의:**
    *   **위험/Gap 상태:** 경고성 오렌지 계열 (`#FF7F4D`) 및 붉은 계열 대비.
    *   **해결/Mitigation 성공:** 황금빛 강조(`Gold Accent: #FFC300` 또는 `#B8860B`). 연결 과정에 빛의 흐름(Bloom) 효과 적용.
3.  **데이터 시각화:** 텍스트 나열 지양. 모든 수치는 그래프, 다이어그램 형태로 변환하여 제시.

---

### 🚀 Step 1: Pain Point / 정보 단절 발견 (The Warning State)
사용자가 현재 가지고 있는 파편화된 지식(Node A와 Node B)이 실제 비즈니스 프로세스에서 어떻게 연결되어야 하는지 질문을 던지는 단계입니다. 시스템은 즉시 Gap의 존재를 경고합니다.

**[컴포넌트명]** $\text{Potential Loss Alert Modal}$
*   **Trigger:** 사용자가 관련 지식 A와 B를 입력했으나, 필수적인 중간 연결 프로세스(Edge)가 누락되었을 때.
*   **UI 구성:** 중앙에 경고 메시지창이 팝업되어 전경을 가립니다. 배경의 그래프는 흐릿하게 보입니다.

| 요소 | 속성 (Props/Data) | 시각적 구현 규칙 | 비고 |
| :--- | :--- | :--- | :--- |
| **제목** | `Potential Disruption Detected` | 대문자, 굵게. 배경 전체에 걸쳐 오렌지색 경고 바를 배치. | 불안감 유발. |
| **핵심 메시지** | "현재 정보 단절로 인한 잠재적 리스크가 식별되었습니다." | Gap의 원인(Process Missing)을 명확히 지목. | 추상적인 설명 X. |
| **손실액 표시 영역** | $\mathbf{X = [계산된 총 잠재 손실액]}$ (예: 억 원대 금액) | 가장 크고, 강렬한 대비를 가진 글꼴로 표시. 배경에 미세하게 떨림(Flicker) 애니메이션 적용. **금전적 손실을 시각적으로 압박.** | *Business/Researcher 데이터 통합 지점.* |
| **요구 조치 버튼** | `[EPHA로 연결 과정 분석하기]` | 이 버튼을 누르면 Step 2 (Connection Flow)로 전환되도록 설계. | 사용자의 다음 행동 유도(CTA). |

---

### ✨ Step 2: Connection Process / 정보 단절 해소 프로세스 (The Discovery State)
사용자가 EPHA의 지능형 엔진을 통해 Gap을 메우는 과정을 실시간으로 시각화합니다. 이 단계에서는 '흐름' 자체가 제품의 핵심 가치가 됩니다.

**[컴포넌트명]** $\text{Dynamic Knowledge Flow Graph}$
*   **Trigger:** Step 1에서 버튼 클릭 후, 시스템이 내부 프로세스 분석을 시작하는 순간.
*   **UI 구성:** 배경에 노드와 연결선(Edge)이 배치된 인터랙티브 그래프가 나타납니다.

| 요소 | 속성 (Props/Data) | 시각적 구현 규칙 | 비고 |
| :--- | :--- | :--- | :--- |
| **현재 Gap 영역** | $\text{Gap Component: 노드 A} \rightarrow \text{[?]} \rightarrow \text{노드 B}$ | 연결선(Edge)의 해당 부분이 끊겨있거나, 불안정한 붉은색 점선으로 표시됨. 이 부분을 중심으로 빛이 부족한 느낌을 강조. | 문제 정의 시각화. |
| **분석 과정 애니메이션** | $\text{Process Step: } \text{Analyze} \rightarrow \text{Map} \rightarrow \text{Connect}$ | 그래프 전체에 은은하고 빠르게 움직이는 그리드(Grid Overlay)와 빛의 파동(Wave Effect)이 오버레이됨. (딥테크 느낌 강화). | '지능적인 작업'을 시각화. |
| **핵심 프로세스 제시** | $\text{Missing Edge: } \text{[규제 프레임워크 매핑] / [필수 검토 절차]}$ | 팝업 또는 사이드 패널에 명확하게 정의된 '정보 연결 과정(Process)'의 단계를 순서대로 나열. (Step-by-step 로직 강조). | 학술적 깊이 유지. |
| **진행률 바** | $\text{Progress: } \text{[1/5 단계 완료]...}$ | 명확한 진행 막대와 함께, 현재 분석 중인 노드를 중심으로 빛나는 에너지가 집중되는 효과를 연출. | 과정에 대한 신뢰도 부여. |

---

### 💰 Step 3: Resolution / 가치 전환 증명 (The Aha Moment State)
Gap이 성공적으로 해소되고, 시스템이 제시한 프로세스를 통해 잠재적 손실액($X$)이 구체적인 절감 가치(Mitigated Value)로 바뀌는 순간을 극대화합니다.

**[컴포넌트명]** $\text{ROI Outcome Dashboard}$
*   **Trigger:** 모든 분석 단계가 완료되고, 최종적으로 'Optimal Path'가 발견되는 순간. (강렬한 시각적 전환 필요).
*   **UI 구성:** 그래프의 어둠과 불안함이 사라지고, 전체 화면에 황금빛으로 빛이 퍼져나가는(Bloom) 트랜지션 효과를 사용합니다.

| 요소 | 속성 (Props/Data) | 시각적 구현 규칙 | 비고 |
| :--- | :--- | :--- | :--- |
| **최종 연결 경로** | $\text{Optimal Path: A} \rightarrow \mathbf{E'} \rightarrow \text{B}$ | Gap이 메워진 부분이 가장 밝은 황금색으로 하이라이트되며, 이로 인해 그래프 전체가 안정되고 완성된 느낌을 줌. | 문제 해결의 명쾌한 증거. |
| **성과 요약 컴포넌트** | $\text{Total Risk Mitigated: } \mathbf{[수치화된 절감 가치]}$ (예: $3억 원 / 연간 운영 비용 감소) | 가장 크고, 중앙에 배치하며, '절감'이라는 행위에 초점을 맞춘 굵은 글씨체 사용. **금전적 결론을 명확히 제시.** | *핵심 메시지.* 잠재 손실액 $\mathbf{X}$가 절감 가치로 전환되었음을 강조. |
| **ROI 상세 계산기** | $\text{ROI Breakdown: } \frac{(\mathbf{X} - \mathbf{Y})}{C_{ 도입}} = Z\%$ | Business의 ROI 공식을 시각적으로 분해하여 보여주는 간결한 차트/도표 형태로 제시. 수식 자체를 디자인 요소화함. | 객관적 근거 제공 (신뢰성). |
| **후속 조치 CTA** | `[이 프로세스 기반으로 개발 로드맵 만들기]` | 다음 단계의 행동을 유도하며, 시스템에 대한 의존도를 높임. | 서비스 확장 및 깊은 몰입 유도. |

**(목표: 정보 단절 해소 프로세스의 가치 증명)**

CEO님의 지시에 따라, 현재까지 논의된 모든 자원(디자인, 스토리보드, 리스크 분석 자료)을 통합하여 가장 빠르고 명확하게 가치를 입증할 수 있는 MVP 범위를 아래와 같이 재정의합니다. 복잡한 데이터베이스 구축보다 **'프로세스 경험'**에 집중하여 초기 개발 난이도와 시장 검증 속도를 극대화하는 것이 핵심입니다.

### 1. ✨ 핵심 기능 정의: 'Gap Path Tracer (정보 연결 경로 추적기)'
MVP의 모든 초점은 사용자가 A라는 지식(Node)에서 B라는 목표(Node)로 가고자 할 때, 현재 어떤 논리적/법적 단절(Gap)이 존재하는지 **단계별 질문**을 통해 발견하고 스스로 해소하는 과정에 맞춰집니다.

*   **MVP 핵심 기능:** 사용자가 두 개의 주요 개념 (A: 현황 지식 / B: 목표 상태)을 입력하면, 시스템은 즉시 답을 주지 않고, 그 사이에 필요한 논리적/규제적 '필수 연결 고리(Bridge)'를 3~5단계의 질문 또는 체크포인트로 분해하여 제시합니다.
*   **작동 방식:** (Input A $\rightarrow$ Gap Detection $\rightarrow$ Step 1 Question $\rightarrow$ User Answer $\rightarrow$ Step 2 Question ... $\rightarrow$ Connection Made).

### 2. 🎯 MVP Scope 한정 및 우선순위 (개발 제한 범위)

| 영역 | 내용 | 개발 난이도 | 비고 (Scope Limiter) |
| :--- | :--- | :--- | :--- |
| **핵심 가치** | 정보 단절을 발견하고, 해소 과정(Process)에 대한 학습 경험 제공. | 중하 | *지식 자체*가 아닌 *질문/단계 구조*가 결과물임. |
| **서비스 범위 제한 (Vertical)** | **의료 기기 (MedTech) 시장의 국내외 규제 준수 프로세스**로 단일화합니다. | 낮음 | 초기 MVP는 가장 법적 리스크와 높은 비용 절감 효과를 수치로 증명할 수 있는 분야에 집중합니다. |
| **데이터 제한** | 전 세계 지식 DB가 아닌, 선정된 가상 시나리오 기반의 *규제 체크리스트* 셋(Set)만 사용합니다. | 낮음 | Mock Data와 구조화된 플로우 차트 형태로 구현하여 데이터 구축 리스크를 제거합니다. |

## 2. 아키텍처 다이어그램 및 데이터 플로우

### [Flow Diagram: `analyzeConnection` 프로세스]

1.  **Frontend (Client):** 사용자 A(현황)와 B(목표) 입력 $\rightarrow$ `POST /api/v1/analysis/start` 호출.
2.  **API Gateway (FastAPI):** 요청 수신 및 세션 관리 $\rightarrow$ 상태 코드 202 Accepted 반환.
3.  **Process Orchestrator (Backend Logic):**
    *   (A, B)를 Graph DB에서 초기 노드 검색 (`Concept` Node).
    *   **[AI Module Call]** : A와 B의 차이를 추론하는 프롬프트 생성 $\rightarrow$ LLM에 전송.
    *   LLM은 Gap 영역을 정의하고 (예: "국가별 인허가 절차") 1단계 질문과 필요한 핵심 키워드(Constraint)를 반환.
4.  **Graph DB Query:** 반환된 Key/Concept를 기반으로 Neo4j에서 연결 가능한 다음 노드를 검색 (`Query A $\rightarrow$ Step 1`).
5.  **Response:** Process Orchestrator는 현재 단계의 질문, 가이드라인(Constraint), 그리고 UI에 필요한 구조화된 JSON 데이터를 Frontend로 전송 (WebSocket 또는 Polling).

### [기술적 상세 설명: 역할 분리]

*   **Graph DB (Neo4j):** **사실 관계와 연결성 (Fact/Structure)** 저장. "A가 B를 필요로 한다"는 구조를 저장합니다.
*   **AI Module (LLM):** **추론 및 질문 생성 (Reasoning/Narrative)** 담당. 데이터만으로는 알 수 없는 '왜 이 단계가 필요한지'에 대한 논리적 근거와 사용자 친화적인 질문을 생성합니다.

## 3. 데이터베이스 스키마 정의 (Schema Definition)

핵심은 **`REQUIRES_STEP`**라는 에지(Edge)를 통해 관계의 *과정*을 모델링하는 것입니다.

### A. Neo4j Graph DB Schema (Primary)

| 노드 타입 (Label) | 속성 (Properties) | 설명 |
| :--- | :--- | :--- |
| `Concept` | `id: string`, `name: string`, `type: ('Current'|'Target')`, `description: string`, `citation_source: string[]` | 사용자가 입력한 핵심 지식 개념 (A, B). |
| `RegulatoryArea` | `id: string`, `name: string`, `scope: string`, `risk_level: integer` | 산업 및 법적 규제 영역 (예: MedTech Class III). |
| `Checkpoint` | `id: string`, `step_number: integer`, `question_text: string`, `required_data: string[]`, `failure_reason: string` | Gap을 해소하기 위해 반드시 거쳐야 하는 단계별 질문 또는 검증 포인트. |

**Edge (관계) 구조:**

| Edge 타입 | 시작 노드 $\rightarrow$ 끝 노드 | 속성 (Properties) | 설명 |
| :--- | :--- | :--- | :--- |
| `IS_RELATED_TO` | `Concept` $\rightarrow$ `RegulatoryArea` | `strength: float`, `details: string` | 기본적인 관계 정의. |
| **`REQUIRES_STEP`** | `Concept(A)` $\rightarrow$ `Checkpoint` | **`step_order: integer`, `logic_description: string`** | A에서 B로 가기 위해 반드시 거쳐야 하는 과정적 단계를 연결함 (MVP 핵심). |
| `IS_RESOLVED_BY` | `Checkpoint` $\rightarrow$ `Concept(B)` | `gap_type: string` | 해당 Checkpoint를 통과함으로써 해소되는 Gap의 종류. |

### B. Supporting DB Schema (PostgreSQL/Redis)

*   **`session_log` (Postgres):**

    *   `user_id`: INT

## 5. 로컬 AI 모듈 연동 설계 (The Brain)

AI는 단순히 정보를 찾는 것이 아니라, **정보 Gap의 논리적 원인을 추론**하여 질문을 생성합니다.

1.  **입력:** Concept A (현황), Concept B (목표).
2.  **Prompt Engineering (핵심):** LLM에게 "A와 B를 연결하는 과정에서 발생하는 가장 큰 법규/기술적 리스크 3가지를 찾아내고, 이를 해소하기 위한 순차적인 질문(Questionnaire)을 작성해줘."라는 구체적인 페르소나 기반의 프롬프트를 제공해야 합니다.
3.  **LLM Output Parsing:** LLM은 구조화된 JSON 형식으로 다음 내용을 반환하도록 강제합니다: `{"gap_reason": "...", "required_steps": [{"step": 1, "question": "...", "keywords": ["..."]}, ...]}`.
4.  **DB Writeback:** 이 구조화된 데이터를 Graph DB의 초기 `Checkpoint` 노드로 저장하고, `REQUIRES_STEP` Edge로 연결합니다. (이 과정이 바로 '정보 단절 해소 프로세스'를 구축하는 핵심 자동화입니다.)

## 📄 EPHA B2B ROI/KPI 계산 로직 확정 문서 (MedTech 특화)

본 문서는 개발팀과 디자인팀이 '정보 단절 해소 프로세스'를 시각적, 정량적으로 증명하기 위한 핵심 재무 모델을 제공합니다. 모든 수치는 **공개된 산업 평균치 및 논리적 추정을 기반으로 하며, MVP 단계에서 테스트 가능한 수준의 가설 값**입니다.

### Ⅰ. 핵심 개념 정의 및 변수 (Input Variables)

| 변수명 | 기호 | 설명 | 데이터 출처/산정 근거 | 단위 |
| :--- | :--- | :--- | :--- | :--- |
| **정보 단절 발생 빈도** | $F_{Gap}$ | 특정 프로세스에서 정보가 끊어지는 예상 주기 (예: 5년마다). | MedTech 시장 리스크 데이터 및 컨설팅 보고서 기반 추정. | Cycle / Year |

| **운영 연속성 비용** | $C_{Downtime}$ | 시스템 오류나 규제 미준수로 인해 서비스가 중단될 때 발생하는 월별 최소 수익/운영 손실액. | 산업 표준화된 병원 운영비(Hospital Operation Cost). | \$ / Month |

### Ⅱ. 계산 공식 로직 (The Formulas)

#### 1. 잠재적 손실액 ($Potential Loss$)
잠재적 손실은 '정보 단절'이 발생할 확률과 그로 인해 발생하는 모든 비용의 합계입니다.

$$\text{Potential Loss} = \left(F_{Gap} \times \text{AvgCost}_{Incident}\right) + (\text{운영 기간 중 예상되는 } C_{Downtime})$$

*   **[개발/디자인 반영 지침]:** 사용자가 Gap을 발견하는 순간, 이 공식에 따른 **'현재까지의 누적 손실액 (Total Potential Loss)'**이 경고창으로 표시되어야 합니다.

#### 2. 최소화된 절감 가치 ($Mitigated Value$)
절감 가치는 잠재적 손실을 EPHA 솔루션의 '정보 연결 과정(Edge)' 구축을 통해 감소시킴으로써 얻는 재정적 이익입니다. 이는 단순히 Gap을 메우는 것을 넘어, **프로세스 자체를 최적화**한 비용입니다.

$$\text{Mitigated Value} = \left(\text{Potential Loss} \times \frac{\eta}{C_{Process}}\right) - (\text{EPHA 도입 비용})$$

*   **[개발/디자인 반영 지침]:** 사용자가 EPHA의 프로세스 추적 과정을 완료할 때마다, '잠재적 손실액'이 수치적으로 감소하고, 그만큼 **'절감된 가치 (Mitigation Credit)'**가 쌓이는 애니메이션을 구현해야 합니다.

### Ⅲ. MedTech 핵심 리스크별 정량화 예시 (3 Case Study)

다음 세 가지 시나리오는 EPHA의 MVP가 가장 강력하게 증명할 수 있는, **정량적 손실액이 큰** 영역입니다.

#### 🏥 1. 규제 준수 정보 단절 리스크 (Compliance Gap)
*   **시나리오:** 환자 기록 데이터(EHR)가 여러 시스템에 분산되어 있어 GDPR/HIPAA 등의 법규상 '정보의 연결성'을 증명하지 못할 경우 발생하는 위반 위험.
*   **$F_{Gap}$ (빈도):** 5년마다 (최소한의 정기 감사 주기 기준).
*   **$\text{AvgCost}_{Incident}$ (사건당 손실액):** **\$1,200,000** (평균 벌금 + 법률 자문 비용)
*   **$C_{Downtime}$ (운영 중단 리스크):** $30만/월 $\times 1.5$개월 = **\$450,000**
*   **$\text{Potential Loss}$:** $(5 \text{년} / 5 \text{년}) \times \$1.2M + \$450K \approx \mathbf{\$1.65 \text{ Million}}$

#### 🧬 2. 진단 과정 데이터 단절 리스크 (Diagnostic Gap)
*   **시나리오:** 환자의 검사 결과(Lab Data), 임상 기록(Doctor's Note), 유전체 정보(Genetic Profile)가 제때 연결되지 않아 초기 진단이 지연되는 경우 (예: 7일 지연).
*   **$\text{AvgCost}_{Incident}$ (사건당 손실액):** **\$150,000** (추가 치료 비용 및 환자 불편에 대한 책임 증대)
*   **$C_{Downtime}$:** $10만/일 $\times 7$일 = **\$700,000** (진료실 가동률 저하로 인한 손실)
*   **$\text{Potential Loss}$:** $(1 \text{회} / 2 \text{년}) \times \$150K + \$700K \approx \mathbf{\$850,000}$

#### 🛠️ 3. 의료 장비 연동/프로세스 단절 리스크 (Integration Gap)
*   **시나리오:** 구형/서로 다른 제조사의 MedTech 장비(예: MRI-PACS)에서 나온 데이터를 수작업으로 통합하는 과정이 필요하여 발생하는 시간적 손실 및 오류.
*   **$\text{AvgCost}_{Incident}$ (사건당 손실액):** **\$3,500 /주** (숙련된 간호사/기술자 1명이 주당 10시간 투입 시의 기회비용)
*   **$C_{Downtime}$:** $2만/일 \times 5$일 = **\$100,000** (데이터 오류로 인한 재작업 비용)
*   **$\text{Potential Loss}$:** $(\text{연간 } 52 \text{주}) \times \$3.5K + \$100K \approx \mathbf{\$288,000}$

        const savingsRate = 0.65;

        setIsGapDetected(true);

    1.  **중앙 그래프 (The Gap):** 노드 A와 B가 명확히 떨어져 있는 지점을 강조합니다. 이 간격(Gap)이 **'정보 단절 영역'**임을 시각적으로 표시하고, 주변에 흩어지고 불안정하게 움직이는 작은 빨간색 데이터 입자들(Data Noise)을 배치합니다.
    2.  **경고 위젯 (Loss Alert):** 화면 상단 또는 좌측에 고정된 **`Potential Loss: $X`** 타이틀의 경고창 컴포넌트를 구현합니다.
        *   **시각 효과:** 배경색은 어두운 빨간색 계열이며, 숫자는 크기(Font Size)와 깜빡임 애니메이션(Blinking Animation)을 적용하여 긴급성을 극대화해야 합니다.
        *   **텍스트 메시지:** "Warning: Critical Information Gap Detected." (Gap의 위험성 강조).

#### 🔵 Stage 2: EPHA 프로세스 개입 및 분석 과정 (The Process Edge)
*   **상태:** 사용자가 'EPHA Process' 버튼 클릭 $\rightarrow$ 애니메이션 실행.
*   **시각적 목표:** 혼돈 속 질서 부여, 지능적인 흐름의 발견(Aha Moment 전 단계).
*   **UI 구성 요소 및 인터랙션 (애니메이션 필수):**
    1.  **Gap 연결 과정 시각화:** 흩어지던 빨간색 노이즈 입자들이 갑자기 **밝은 시안 블루 빛의 흐름(Edge)**을 따라 모여들기 시작합니다. 이 Edge는 단순한 선이 아니라, 여러 개의 작은 '연결 프로세스' 컴포넌트들이 순차적으로 작동하는 듯한 애니메이션이어야 합니다.
    2.  **정보 흐름 다이어그램:** Gap 주변에 마치 레이더 스캔처럼 **그리드 오버레이(Grid Overlay)**가 겹치며, EPHA가 어떤 각도와 방법으로 정보를 연결하고 있는지 기술적인 신뢰도를 높이는 시각적 장치를 추가합니다.
    3.  **분석 리포트 패널:** 화면 한쪽에 슬라이드아웃 되는 'Analysis Panel'을 배치하여, "EPHA Analysis Report: Connection Process Required"라는 제목과 함께 **정보 단절의 원인(Gap Reason)** 및 필요한 로직 구조를 간결하게 텍스트와 다이어그램으로 제시합니다.

#### 🟢 Stage 3: 절감 가치 발견 (The Mitigation & ROI)
*   **상태:** 프로세스 분석 완료 $\rightarrow$ 결과값 계산 및 표시.
*   **시각적 목표:** 명확한 해답, 해결책의 구조화된 증명, 경제적 이득의 확신 부여.
*   **UI 구성 요소:**
    1.  **재정적 가치 전환 (The Switch):** Stage 1의 빨간색 'Potential Loss' 경고창이 **부드럽게 축소되며 사라지고**, 그 자리에 골드 그린 테두리의 새로운 패널이 등장합니다.
    2.  **핵심 KPI 대시보드:** 다음 수치가 가장 크고 눈에 잘 띄는 타이포그래피로 표시되어야 합니다.
        *   `Mitigated Value (Expected Savings): $Y$`
        *   `ROI Improvement: Z%`
    3.  **최종 그래프:** A와 B 노드를 연결하는 Edge가 이제 불안정한 파란색이 아닌, **견고하고 안정적인 골드 그린 색상의 구조적 경로**로 확정되어 표시됩니다. 이 과정은 '데이터 흐름'이 아니라 '확보된 자산(Asset)'처럼 보이게 디자인해야 합니다.
