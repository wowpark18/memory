### 🚀 상위 티어 업그레이드 핵심 판매 포인트 (The Upgrade Narrative)

각 등급에서 다음 단계로 이동해야 하는 이유(Pain Point 해결 및 가치 증명)를 강조합니다.

#### 💡 Basic $\rightarrow$ Pro: "단순히 무엇이 부족한지 찾는 것을 넘어, *시스템 자체*를 개선하세요."
1.  **다중 변수 분석 (Multi-Variable Analysis):** Basic은 A와 B의 비교만 하지만, Pro는 '사람', '시간', '규정' 등 3가지 이상의 변수를 동시에 투입하여 손실을 진단합니다.
2.  **행위 기반 추적 (Behavioral Tracking):** 단순히 문서 Gap을 찾는 것이 아니라, *누가*, *언제*, *어떤 과정*에서 지연되었는지를 프로세스 단위로 시간대별 로그를 남기며 분석할 수 있습니다.
3.  **자동화된 해결책 제안:** 발견된 문제에 대해 "이렇게 수정하세요"라는 컨설팅 보고서와 함께, 즉시 도입 가능한 워크플로우 개선 액션 플랜(Action Plan)을 제공합니다.

#### 💡 Pro $\rightarrow$ Enterprise: "현재의 문제를 고치는 것을 넘어, *미래를 예측*하고 시장 주도권을 확보하세요."
1.  **외부 데이터 통합 (External Data Integration):** 내부 프로세스 자료만 보는 것이 아니라, 거시 경제 지표, 경쟁사 특허 출원 동향 등 외부의 비정형 데이터를 실시간으로 끌어와 $L-Cost$ 계산에 반영합니다.
2.  **예측 모델링 및 시뮬레이션 (Predictive Simulation):** "현재 이대로 가면 6개월 뒤 A 시장에서 $X$만큼 손해 볼 것입니다"와 같이, 가설과 변수를 기반으로 미래의 위험도를 확률적으로 제시하고 최적화된 대응 경로를 설계합니다.
3.  **전사적 거버넌스 연동 (Enterprise Governance):** 전사의 핵심 시스템(ERP, CRM 등)에 EPHA 엔진을 직접 API로 연결하여, $L-Cost$ 진단이 회사의 의사결정 프로세스 자체에 내재화됩니다.

```javascript
const calculateAndStreamLoss = (inputData, socket) => {

                message: "최대 위험 감지! 즉각적인 전략적 개입이 필요합니다.",
```

```javascript
// LCostDisplayComponent.jsx

const [lossState, setLossState] = useState({ cost: 0, isCritical: false });

useEffect(() => {
    if (socket && socket.readyState === WebSocket.OPEN) {
        socket.onmessage = (event) => {
            const data = JSON.parse(event.data);
            
            // 상태 업데이트 로직 실행
            setLossState({ cost: data.lcost, isCritical: data.isCritical });

            if (data.type === 'CRITICAL_ALERT') {
                triggerRedPulseAnimation(); // Red Pulse 애니메이션 함수 호출
            }
        };
    }
}, [socket]);


// 핵심 애니메이션/스타일 로직
const getDisplayStyles = () => {
    let color;
    if (lossState.isCritical) {
        color = 'rgb(200, 5, 5)'; // Red Pulse Color Code
        return { backgroundColor: color, boxShadow: '0 0 30px rgba(200, 5, 5, 0.8)' };
    } else if (lossState.cost > 50) {
        color = 'rgb(255, 165, 0)'; // Warning Orange
        return { backgroundColor: color, boxShadow: '0 0 20px rgba(255, 165, 0, 0.5)' };
    } else {
        color = 'rgb(76, 175, 80)'; // Green Safe Color
        return { backgroundColor: color, boxShadow: 'none' };
    }
};

// CSS/Tailwind class를 동적으로 적용하는 로직이 필요함.
```

### 📑 B. 본문 요약 버전 (Short Body Copy - Landing Page 메인 섹션)

**[문제 제기: Pain Point]**
최고 경영진에게 중요한 것은 '무엇이 부족한가'라는 질문이 아닙니다. 가장 중요한 것은 **'어디서 돈을 잃고 있는가'**입니다. 수많은 보고서와 데이터 속에서, 조직은 종종 다음과 같은 형태로 손실을 안고 살아갑니다.
1.  **$\text{L}_{Info}$ (정보 손실):** 핵심 트렌드나 경쟁사 변화에 대한 정보가 특정 부서/문서에만 국한되어 발생합니다.
2.  **$\text{L}_{Process}$ (프로세스 손실):** 승인 단계의 불필요한 병목 현상, 반복적인 재작업 등으로 자원이 비효율적으로 소진됩니다.

### I. L-Cost 측정 표준화 템플릿 (The Quantification Engine)

모든 산업의 손실은 단일한 원인이 아닙니다. $L_{Total}$은 세 가지 상호 연관된 요소로 분해되어야 하며, 각 요소별로 정량적인 ROI 산출이 가능해야 합니다.

$$ L_{\text{Total}} = L_{\text{Info}} + L_{\text{Process}} + L_{\text{Strategy}} $$

| 손실 유형 | 정의 (Gap의 종류) | 측정 대상 지표 (KPI) | 재무적 근거 산정 방식 |
| :--- | :--- | :--- | :--- |
| **$L_{Info}$** (정보 부족 손실) | 시스템/데이터 사일로, 예측 모델 부재 등 *지식의 연결 단절*에서 오는 위험. | **미발견 리스크 건수 / 정보 검색 평균 시간($T_{search}$) 증가율.** | $L_{\text{Info}} = (\text{평균 단위 비용}) \times (\frac{\text{데이터 사일로 발생 비율}}{\text{정보의 가치 밀도}}) \times (1 + \text{시간 지연 계수})$ |
| **$L_{Process}$** (프로세스 비효율 손실) | 수작업, 병목 현상(Bottleneck), 부서 간 협업 오류 등 *절차적 낭비*에서 오는 위험. | **재작업 비율(Rework Rate) / 프로세스 단계별 평균 소요 시간($T_{step}$) 증가율.** | $L_{\text{Process}} = (\text{인건비 및 자원 비용}) \times (\text{불필요한 순환 횟수}) + (\text{재작업으로 인한 추가 공수})$ |
| **$L_{Strategy}$** (전략적 오류 손실) | 시장 변화 예측 실패, 목표 불일치, 핵심 가치 간 괴리 등 *의사결정 구조*에서 오는 위험. | **KPI 달성률 미달로 인한 예상 수익($R_{\text{expected}}$) 대비 실제 수익($R_{\text{actual}}$) 격차.** | $L_{\text{Strategy}} = R_{\text{expected}} - R_{\text{actual}}$ (최대 손실액 기준으로 계산) |

*   **핵심 메시지:** "불확실한 시장 환경에서 의사결정이 무너지지 않도록, 구조적 데이터를 기반으로 최적의 대응 시나리오를 제공합니다."

**목표:** $L_{Total} = L_{Info} + L_{Process} + L_{Strategy}$를 실시간으로 계산하고, 위험 상태 전환(Warning $\rightarrow$ Critical $\rightarrow$ Solution)을 API로 처리하는 백엔드 로직 정의.
**기술 스택 가이드:** Python (FastAPI/Flask 기반 구조 가정), TypeScript (타입 안정성 확보).

### 1. 데이터 모델 및 초기화 스키마 (`Schema.ts` / `data_model.py`)

시스템이 처리해야 할 모든 상태 데이터를 정의합니다.

```typescript
// State Schema Definition for the Simulation Engine

interface InitialInput {
    userIndustry: string;         // 예: Manufacturing, Finance 등 (산업군)
    currentOperationalScore: number; // 초기 운영 점수 (0-100점). 높을수록 좋음.
    dataIntegrityLevel: number;  // 정보 무결성 레벨 (0-100%). L_Info의 기반 데이터.
    processEfficiencyRating: number;// 프로세스 효율성 등급 (0-10). L_Process의 기반 데이터.
    strategicVisibilityScore: number;// 전략적 가시성 점수 (0-100점). L_Strategy의 기반 데이터.
}

interface SimulationState {
    stateId: string;
    timestamp: Date;
    status: 'Normal' | 'Warning' | 'Critical' | 'Resolved'; // 현재 상태 정의
    metrics: InitialInput;
    lInfo: number;     // 정보 단절 손실 (L_Info)
    lProcess: number;  // 프로세스 비효율 손실 (L_Process)
    lStrategy: number; // 전략 부재 손실 (L_Strategy)
    lTotal: number;    // 총 손실액 (L_Total = L_Info + L_Process + L_Strategy)
    recommendation: string; // 엔진의 최종 조언/해결책 방향
}
```

### 2. 핵심 로직 구현 (`SimulationEngineService.py`)

$L_{Total}$을 계산하는 비즈니스 로직 코어입니다. 이 함수가 시스템의 가장 중요한 부분이며, 모든 API 호출은 이 함수를 거쳐야 합니다.

```python
# Python Pseudo-Code for the core calculation logic (SimulationEngineService)

def calculate_l_total(state: InitialInput) -> tuple[float, str]:
    """
    입력된 상태 데이터를 기반으로 3가지 손실을 계산하고 총합과 다음 상태를 예측합니다.
    L_Cost는 가중치와 임계값 로직이 포함되어야 합니다.

    Args:
        state (InitialInput): 현재 시스템의 측정 데이터.

    Returns:
        tuple[float, str]: 계산된 L_Total 값과 엔진이 판단한 상태 메시지.
    """
    # 1. 가중치 및 손실 공식 정의 (가정적 로직)
    # L_Info는 dataIntegrityLevel에 역비례합니다.
    L_info = max(0, 50 - state['dataIntegrityLevel'] * 0.2)

    # L_Process는 operationalScore와 processEfficiencyRating의 결합으로 측정됩니다.
    process_gap = (100 - state['currentOperationalScore']) / 10.0
    L_process = max(5, process_gap + (10 * (10 - state['processEfficiencyRating'])))

    # L_Strategy는 strategicVisibilityScore가 낮을수록 급증합니다.
    L_strategy = max(0, 100 - state['strategicVisibilityScore']) / 2.0

    # 2. 총 손실액 계산 및 임계값 적용 (Link Strength 지수 반영)
    l_total = round(L_info + L_process + L_strategy, 2)

    # 3. 상태 판단 로직 (Critical Trigger 정의)
    if l_total >= 150:
        status = "CRITICAL"
        recommendation = "즉각적인 Gap 분석 및 자원 재배치가 필요합니다. 핵심 프로세스 중단 위험."
    elif l_total >= 80:
        status = "WARNING"
        recommendation = "위험 신호 포착. 정보 단절 영역을 중심으로 추가 진단을 진행해야 합니다."
    else:
        status = "NORMAL"
        recommendation = "현재 상태는 안정적이나, 전략적 가시성 확보를 통해 잠재 위험을 제거해야 합니다."

    return l_total, status + ", " + recommendation


def generate_initial_state(input_data: InitialInput) -> SimulationState:
    """초기 입력을 받아 최초의 시뮬레이션 상태 객체를 생성합니다."""
    l_total, status_message = calculate_l_total(input_data)

    return {
        "stateId": f"SIM-{int(time.time())}",
        "timestamp": datetime.now(),
        "status": "Normal", # 초기 진입 시점은 일단 Normal로 설정하고, 실제 계산된 상태를 반영해야 함 (API 호출부에서 수정 필요)
        "metrics": input_data,
        "lInfo": round(150 - 50 * input_data['dataIntegrityLevel'] / 100, 2), # 예시 값
        "lProcess": round(30 + (input_data['processEfficiencyRating'] * 2), 2),
        "lStrategy": round(100 - input_data['strategicVisibilityScore'], 2),
        "lTotal": l_total,
        "recommendation": status_message
    }

```

### 3. API 엔드포인트 정의 (`FastAPI/Flask` Mockup)

실제 시스템과의 연동을 위한 세 가지 필수 API 엔드포인트를 정의합니다. (HTTP 메서드는 명시적입니다.)

#### A. 초기 진단 및 상태 설정
**Endpoint:** `POST /api/v1/simulate/initialize_state`
**설명:** 사용자의 산업군, 현재 점수 등을 입력받아 최초의 $L_{Total}$을 계산하고 시뮬레이션 시작점을 확립합니다.
**Request Body (JSON):** `{ "userIndustry": "Finance", "currentOperationalScore": 75, "dataIntegrityLevel": 60, "processEfficiencyRating": 8, "strategicVisibilityScore": 40 }`
**Response Body (200 OK - JSON):** `SimulationState` 객체 전체 반환.

#### B. 실시간 위험 감지 및 상태 전환 트리거
**Endpoint:** `POST /api/v1/simulate/check_risk`
**설명:** 시간 경과에 따라 변하는 특정 데이터 포인트(예: 마케팅 지출 대비 매출 감소율)를 입력받아, $L_{Total}$의 변화와 상태 전환을 감지합니다. **이 엔드포인트가 'Critical' 진입을 트리거합니다.**
**Request Body (JSON):** `{ "stateId": "SIM-XXXX", "updated_metric": "SalesRatio", "new_value": 0.4 }`
**Response Body (200 OK - JSON):** `SimulationState` 객체, 특히 `status` 필드와 변경된 `lTotal` 값을 강조하여 반환합니다.

#### C. 해결책 제시 및 상태 복구 업데이트
**Endpoint:** `POST /api/v1/simulate/update_solution`
**설명:** 컨설팅 또는 내부 조치를 통해 문제를 해결한 후, 개선된 데이터를 입력받아 $L_{Total}$이 감소하는 과정을 시뮬레이션하고 'Resolved' 상태로 전환합니다.
**Request Body (JSON):** `{ "stateId": "SIM-XXXX", "improvement_action": "Process Reengineering", "new_metrics": { "dataIntegrityLevel": 85, "processEfficiencyRating": 9, "strategicVisibilityScore": 70 } }`
**Response Body (200 OK - JSON):** `SimulationState` 객체. 성공적으로 손실이 감소하고 상태가 개선되었음을 명확히 보여줍니다.

*(A Proactive Framework for Financial Value Protection: $L-Cost$ Based Business Risk Management)*

### I. 서론: C-Level이 직면한 그림자 비용 (The Unseen Cost)
**[목표] 문제 제기 및 공감대 형성. 독자가 스스로 '무언가 중요한 것을 놓치고 있다'는 불안감을 느끼게 한다.**
*   **A. 시대적 변화와 리스크의 복잡성:** 산업 구조의 급변 속도, 데이터 사일로화 심화에 따른 비즈니스 운영의 난이도 증대. (현상 제시)
*   **B. 전통적 리스크 관리의 한계:** 기존 재무/운영 감사(Audit)는 '이미 발생한' 손실을 측정하는 사후 대응적 방식에 머무름. 예측 불가 영역($L_{Total}$)은 포착 불가능. (기존 방법 비판)
*   **C. 핵심 질문 제기 (Hook):** "우리가 놓치고 있는, 가장 큰 재정적 기회비용은 무엇인가?"

### II. 이론적 배경: 지식 단절과 시스템 실패의 경제학 (The Gap Theory)
**[목표] 학술적 깊이를 갖추되, 이를 '돈'이라는 언어로 번역하여 설명한다.**
*   **A. 정보 비대칭성과 프로세스 사일로:** 데이터가 물리적으로 분리된 것보다, **정보 흐름이 끊긴 곳($L_{Info}$)**에서 발생하는 손실이 더 치명적임. (개념 정의)
    *   *Case Example:* 제조업 PdM 데이터의 여러 단계(센서 $\rightarrow$ 분석 $\rightarrow$ 의사결정) 사이의 정보 단절 비율 측정 필요성 제시.
*   **B. 지식 격차($F_{Gap}$)를 재무적 손실로 연결:** 단순히 '지식이 부족하다'가 아니라, '이 지식 부재로 인해 $X$ 시간/금액을 잃는다'는 식으로 정량화. (논리 전환)
*   **C. EPHA의 핵심 가치: 측정 가능한 재무적 손실 발견 경험($L-Cost$) 정의.**

### III. 방법론 제시: L-Cost 기반 리스크 예측 프레임워크 (The Solution)
**[목표] 구체적인 작동 방식과 단계별 로직을 시각적으로 보여준다. 백서의 가장 중요한 부분.**
*   **A. $L_{Total}$ 측정 모델:** 총 손실액($L_{Total}$) = 직접 비용 + 간접 비용 + **측정 가능한 정보 손실 비용($L-Cost$)**.
*   **B. EPHA 3단계 동적 진단 여정 (Diagnostic Journey):**
    1.  **[Red Pulse] 위험 경고(Warning & Detection):** 시스템의 이상 패턴 감지. "지금 이 영역에서 $X$만큼 손실이 예상됩니다." (선제적 포착)
    2.  **[Analysis] 원인 분석 및 근거 제시:** 단순 알람을 넘어, 왜 이 위험이 발생했는지(데이터 사일로 지점 $\rightarrow$ 프로세스 단절 지점 등)를 구조적으로 추적하고 보고서화. (원인 규명)
    3.  **[Resolution] 해결책 제안 및 ROI 예측:** 재발 방지 워크플로우 구축을 통해 손실액($L-Cost$)을 최소화할 때 얻는 기대 효과(ROI) 제시. (해결과 가치 증명)

### IV. 적용 사례 및 비전 (Proof & Future Vision)

*   **A. 산업별/손실 유형별 성공 사례:** (제조업 PdM 외, 금융/헬스케어 등 다양한 산업군을 예시로 들어 L-Cost 측정 가능성 입증).
*   **B. 하이브리드 수익 모델 제시:** 컨설팅 기반의 초기 $L_{Total}$ 진단 $\rightarrow$ SaaS 기반의 지속적 모니터링 및 개선(Subscription)으로 전환되는 명확한 로드맵.

### 3. 핵심 요소: $L-Cost$ 측정 흐름 플로우차트 (The Signature Diagram)
이 다이어그램은 백서의 모든 페이지에서 개념적으로 반복되어야 하며, EPHA의 존재 이유 그 자체입니다.

**[제목] L-Cost Discovery Cycle: 미발견 위험 포착 3단계 모델**

1.  **Stage 1: Pain Point (위험 인지/정보 사일로)**
    *   **시각 요소:** 복잡하고 산재된 데이터 노드들(Node)이 무질서하게 배치됨. 연결선은 끊기거나 희미함. 배경에 `#D93A2E`의 불안정한 빛을 사용.
    *   **핵심 지표:** $L_{Total}$ (총 손실 가능액)를 빨간색 큰 글씨로 표시.
    *   **서사:** "데이터는 존재하지만, 연결되지 않아 가치가 0에 수렴하는 상태."

2.  **Stage 2: EPHA Intervention (분석/연결 프로세스)**
    *   **시각 요소:** 중앙에 'EPHA Engine'을 상징하는 모듈(Module)이 배치됨. 이 엔진에서 빛나는 `#00AEEF` 계열의 에너지 흐름(Edge)이 터져 나오며 노드들을 연결하기 시작함. 프로세스 경로를 나타내는 투명한 그리드가 오버레이 됨.
    *   **핵심 지표:** $L_{Info}$ (정보 단절 비율) 및 $F_{Gap}$가 측정되어 수치화되는 과정 시각화.
    *   **서사:** "무작위 데이터 속에서 패턴과 관계성을 찾아내고, 손실의 근원을 분석함."

3.  **Stage 3: Resolution (통찰/해결책 제시)**
    *   **시각 요소:** 모든 노드가 하나의 명확한 경로(Optimal Path)로 연결되고, 중앙에 'Key Insight'가 밝게 빛남. 배경 색상이 어두운 네이비에서 신뢰감 있는 청색 계열로 전환됨.
    *   **핵심 지표:** $L_{Cost}$ (측정 가능한 재무적 손실액 발견)를 명확한 녹색 또는 `#00AEEF`의 가장 밝은 색으로 강조하여 제시.
    *   **서사:** "미발견된 위험이 구체적인 해결책과 수익화 기회로 전환되는 경험."

# 🧬 EPHA Core Logic: 데이터 및 이벤트 스펙 정의 (Data & Event Schema)

## 1. 핵심 데이터 모델 (Data Schemas)

진단 프로세스에서 관리해야 할 세 가지 주요 엔티티(DiagnosisSession, DataNode, LossMetric)의 JSON 스키마를 정의합니다.

### A. `DiagnosisSession` Schema (세션 상태 및 메타데이터)
사용자의 현재 진단 여정의 전체적인 맥락과 목표를 담습니다.

```json
{
  "session_id": "uuid-string",         // Unique ID for the current session
  "user_id": "user-identifier",       // Associated user/client
  "start_time": "datetime-iso",
  "current_status": {                  // Current State (e.g., WARNING, CRITICAL)
    "state_name": "string",           // e.g., 'WARNING', 'CRITICAL', 'RESOLVED'
    "severity_level": 0.0              // Numeric score (0.0 to 1.0)
  },
  "diagnosis_history": [               // Array of State Transitions
    {
      "timestamp": "datetime-iso",
      "previous_state": "string",       // e.g., 'INITIAL'
      "triggered_event": "string",     // The event that caused the change (e.g., 'GAP_DETECTED')
      "new_state": "string",            // The resulting state
      "description": "Transition summary message."
    }
  ]
}
```

### B. `DataNode` Schema (데이터 노드 - 지식/정보 단위)
진단 과정에서 발견되거나 입력되는 개별 정보 단위를 정의합니다.

```json
{
  "node_id": "uuid-string",
  "source_system": "string",         // Data Source (e.g., CRM, ERP, Manual Input)
  "topic": "string",                  // The subject matter (e.g., 'Customer Churn', 'Supply Chain Delay')
  "data_value": "any",                // Actual data point (numeric, text, array)
  "confidence_score": 0.8,           // AI/System confidence in this node's accuracy
  "metadata": {                       // 추가 컨텍스트 정보
    "owner": "team-A",
    "last_updated": "datetime-iso"
  }
}
```

### C. `LossMetric` Schema (손실 측정 변수 - $L-Cost$ 구성 요소)
핵심 가치 지표인 손실액을 계산하기 위한 모든 입력 데이터를 모읍니다.

```json
{
  "metric_id": "uuid-string",
  "type": "enum(L_INFO, TIME_DELAY, OPPORTUNITY)", // 손실 유형 분류
  "value": 150000.0,                   // 측정된 금전적 또는 시간적 가치 (단위: USD)
  "unit": "currency/hour/percentage",  // 단위 명시
  "calculation_basis": {               // 이 값이 산출된 근거
    "input_nodes": ["node-id-1", "node-id-2"], // 어떤 노드들로 계산되었는지 추적
    "formula_applied": "L_Info = (T * Gap)" // 사용된 공식 명시
  },
  "risk_severity": 0.75                // 이 손실이 현재 세션에 미치는 위험 점수
}
```

## 2. 이벤트 기반 아키텍처 (EDA) 로직 구조도

진단 여정은 **상태 기계(State Machine)**로 관리되며, 모든 데이터 입력 및 프로세스 변화는 **이벤트**를 통해 트리거됩니다.

| 단계/상태 | 상태 이름 (`current_status`) | 진입 조건 (Transition Event) | 처리 로직 (Service Action) | 출력 결과 (Output / Next State) |
| :--- | :--- | :--- | :--- | :--- |
| **1. 관찰 (Observation)** | `INITIAL` | N/A (Session Start) | 초기 $L-Cost$ 값 및 핵심 변수 정의 ($L_{Total}=0$) | `Initial Diagnostic Report` |
| **2. 데이터 입력** | `ANALYZING` | `DATA_NODE_RECEIVED` | 1. 신규 `DataNode`를 Graph DB에 추가. 2. 연결된 모든 노드 쌍을 탐색 (Graph Traversal). | 업데이트된 $L_{Info}$ 계산 및 그래프 시각화 데이터 반환. |
| **3. 간극 감지** | $\rightarrow$ `WARNING` | `GAP_DETECTED` (Triggered by Graph Logic) | 1. 연결이 끊긴 노드/엣지를 식별. 2. $L_{Info}$를 계산하여 `LossMetric`에 기록. | 경고 발생. 사용자에게 "데이터 간극 발견" 알림. |
| **4. 임계점 도달** | $\rightarrow$ `CRITICAL` | `MULTIPLE_GAPS_DETECTED` + `HIGH_TIME_DELAY` | 1. $L_{Info}$, 시간 지연 비용, 기회비용을 종합 계산하여 최종 $L-Cost$ 산출. 2. 최고 위험 등급으로 상태 변경. | **위험 경고 발생.** 해결책 제시 및 다음 액션 요구 (Intervention). |
| **5. 해결/종료** | `RESOLVED` | `INTERVENTION_APPLIED` / `MANUAL_RESOLUTION` | 시스템이 제안한 해결책을 적용하고, $L-Cost$를 감소시키는 시뮬레이션을 진행. | 최종 진단 보고서 및 ROI 계산 결과 제공. |

## 3. API 엔드포인트 명세 (Swagger Style)

백엔드 백본(Backend Backbone)의 핵심 역할을 수행할 세 가지 필수 엔드포인트를 정의합니다.

### A. `POST /api/v1/diagnosis/start`
*   **기능:** 새로운 진단 세션을 시작하고 초기 상태를 설정합니다.
*   **요청 바디 (Request Body):** `{ "user_id": "string", "initial_context": "string" }`
*   **응답 (Response 201 Created):** `{"session_id": "uuid-string", "status": "INITIAL", "message": "Diagnosis session started successfully."}`

### B. `POST /api/v1/diagnosis/eventstream` **(⭐️ 핵심 이벤트 스트림)**
*   **기능:** 진단 프로세스에 새로운 데이터를 주입하고, 시스템이 이를 감지하여 상태 변화를 트리거하는 중앙 허브입니다. 모든 데이터 입력은 이곳으로 들어옵니다.
*   **요청 바디 (Request Body):** `{ "session_id": "uuid-string", "event_type": "enum(DATA_NODE_RECEIVED, EXTERNAL_API_CALL, MANUAL_INPUT)", "payload": { /* 해당 이벤트의 상세 데이터 */ } }`
    *   *예시 Payload (`DATA_NODE_RECEIVED`):* `{"node_id": "uuid", "source": "CRM", "topic": "Sales", "data_value": 100}`
*   **응답 (Response 200 OK):** `{ "status": "SUCCESS", "current_state": "string", "diagnostics": { /* 현재의 L-Cost 분석 결과와 경고 목록 */ } }`

### C. `GET /api/v1/diagnosis/{session_id}/report`
*   **기능:** 특정 세션의 최종 진단 보고서(L-Cost 총합, 위험 경로 시각화 데이터)를 조회합니다.
*   **경로 매개변수 (Path Parameters):** `session_id`
*   **응답 (Response 200 OK):** `{ "l_total_cost": 987654.32, "currency": "USD", "breakdown": [ /* LossMetric 리스트 */ ], "timeline": [ /* DiagnosisSession.diagnosis_history */ ] }`

## 📂 Case Study 1: '경고 신호'를 놓치는 비용 - 예측 가능성 손실 ($L_{Info}$)
**(Focus: 데이터의 존재 여부가 아닌, 활용되지 못하는 시점 자체에 대한 리스크)**

**[1. 현재 상태와 놓치고 있는 위험 제시 (Hook)]**
귀사의 공장 라인에는 수많은 IoT 센서가 설치되어 실시간으로 기계의 미세한 진동 변화를 감지하고 있습니다. 이는 훌륭합니다. 하지만 이 '경고' 데이터가 설비 운영 계획, 인력 배치 시스템과 같은 핵심 의사결정 흐름에 연결되지 않는 순간이 발생합니다. 마치 고성능 경보기가 울리고 있지만, 그 소리가 수백 개의 다른 업무 알람 속에서 무시되는 것과 같습니다. 귀사는 실제로 기계가 멈추는 시점(Failure Point)에만 대응하고 있으며, **'이상 징후가 감지된 최적의 시간대'**라는 가장 가치 있는 정보를 놓치고 있습니다.

**[2. 손실액 추정 및 측정 가능한 지표 제시 (The Cost)]**
이러한 '정보 단절(Information Gap)'은 단순한 데이터 누락을 넘어, **예측 가능성 상실에 따른 추가 비용($L_{Info}$)**으로 직결됩니다. 가장 큰 재무적 손실은 기계가 멈춘 시간($T_{downtime}$) 자체가 아니라, *그 위험 신호가 발견되었음에도 불구하고 적절한 조치가 취해지지 못하여* 발생한 **'기회 창(Opportunity Window)'의 상실**입니다.
*   **핵심 지표:** Time-to-Detection Gap (TTDG) 분석을 통해, 실제 고장까지 남은 시간과 시스템이 경고를 보낼 수 있었던 최적 시점 사이의 간극을 돈으로 환산해야 합니다.

**[3. EPHA가 제시하는 다음 단계 (The Solution)]**
EPHA는 단지 데이터를 모으는 것이 아니라, **'경보 → 분석 → 조치 계획'의 동적 흐름(Dynamic Workflow)**을 구축합니다. 시스템이 이상 징후를 감지하는 순간, 자동으로 운영팀과 자재 구매 담당자에게 '최소한의 필요 조치 리스트'와 '예상되는 재무적 손실액 감소 시뮬레이션 결과'를 전달하여, 사일로화된 정보를 하나의 실행 가능한 지시(Actionable Directive)로 통합합니다.

## 📂 Case Study 2: 반복되는 실수에 대한 비용 - 프로세스 누락 손실 ($L_{Process}$)
**(Focus: 문제 해결 경험 자체의 기록 및 시스템 학습 실패)**

**[1. 현재 상태와 놓치고 있는 위험 제시 (Hook)]**
기술팀이 현장에서 설비 고장을 성공적으로 수리했다고 보고하는 것만으로는 충분하지 않습니다. 가장 큰 비용은 **'같은 문제를, 또다시 겪는 반복적인 재발 리스크'**에서 발생합니다. 기술자들이 현장에서 얻는 귀중한 노하우와 근본 원인 분석(RCA) 과정이 공식적인 시스템의 '지식 자산'으로 축적되지 못하고, 개별 보고서나 개인 메모리에만 머물러 있습니다. 이처럼 체계화되지 않은 지식은 기업에게 **'반복되는 구조적 무지(Structural Ignorance)'**라는 형태로 비용을 청구합니다.

**[2. 손실액 추정 및 측정 가능한 지표 제시 (The Cost)]**
이 누락된 프로세스 기록은 다음 고장을 예방하는 데 사용될 수 있었던 잠재적 가치를 훼손하며, 이는 **'반복 재발 위험 비용($L_{Process}$)'**으로 계산됩니다. 매년 반복되는 사소한 오류가 모여 쌓인 구조적인 손실을 찾아야 합니다.
*   **핵심 지표:** Mean Time Between Failure (MTBF)의 하락 속도(Degradation Rate)를 추적하세요. 개선된 프로세스 없이 시간이 흐름에 따라 MTBF가 떨어지고 있다면, 귀사는 현재 '지식 자산화 실패'라는 심각한 리스크에 직면해 있습니다.

**[3. EPHA가 제시하는 다음 단계 (The Solution)]**
EPHA는 고장 발생 시점부터 원인 분석(RCA)을 필수 워크플로우로 강제합니다. 단순 수리 보고서 작성을 넘어, '어떤 부품이 왜 망가졌는지'에 대한 **원인-결과 그래프(Cause-Effect Graph)**를 자동으로 생성하고 이를 재고 관리 및 교육 시스템에 피드백하여, 동일한 실패 루프가 발생할 확률 자체를 구조적으로 0으로 수렴시키는 것이 목표입니다.

## 📂 Case Study 3: 시스템 마비 비용 - 통합 단절 손실 ($L_{Info} + L_{Process}$)
**(Focus: 가장 복잡하고 크리티컬하며, 모든 시스템이 분리되었을 때의 총체적 위험)**

**[1. 현재 상태와 놓치고 있는 위험 제시 (Hook)]**
가장 치명적인 상황은, **'모든 것이 제대로 작동하는 것처럼 보이지만, 사실은 연결된 부분이 전혀 없는'** 상태입니다. 예를 들어, 핵심 부품이 곧 부족할 것 같은 경고(IoT)를 받고, 기술팀도 원인을 분석했으나(RCA), 재고 관리 시스템(ERP)과 공급망 정보가 완전히 분리되어 있습니다. 이 경우, 귀사는 '현재의 고장'을 넘어 **'미래에 필요한 자원의 절대적 부재'**라는 치명적인 마비 상태에 빠집니다. 모든 데이터 사일로가 결합된 블랙홀 속에서, 진정한 위험은 측정 불가능한 공포 그 자체입니다.

**[2. 손실액 추정 및 측정 가능한 지표 제시 (The Cost)]**
이 통합 단절은 단순 합산 이상의 문제입니다. 이는 **'시스템 마비 총체 비용($L_{Total}$)'**으로 정의됩니다. 이 비용에는 단순히 부품을 비상 구매한 값(구매가)뿐 아니라, 그 때문에 전체 생산 라인이 멈추면서 발생하는 **인건비 손실, 계약 위약금, 시장 점유율 하락에 따른 기회 상실액**까지 포함해야 합니다.
*   **핵심 지표:** 부품 수급 지연으로 인한 총 가동 중단 시간($T_{Total}$)을 최소 단위(시간/분)로 측정하는 것이 가장 중요합니다.

**[3. EPHA가 제시하는 다음 단계 (The Solution)]**
EPHA는 이 세 가지 단절된 영역(감지 $\rightarrow$ 분석 $\rightarrow$ 공급망)을 **하나의 중앙 지능형 프로세스 엔진**으로 통합합니다. 위험 경고 발생 시, 시스템은 자동으로 '필요 부품 리스트', '가장 빠른 대체 경로', 그리고 '최적화된 비상 조치 로드맵'까지 연쇄적으로 제시하여, 예측만 하는 것이 아니라 **완벽하게 제어 가능한 대응책**을 제공함으로써 재무적 손실의 전 과정을 사전에 차단합니다.
